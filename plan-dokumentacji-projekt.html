Moduł: Wizja i zakres produktu

📐 Product Vision & Scope
Cel: zdefiniować CO budujemy i czym to NIE jest

Ten dokument to "konstytucja" projektu. Każda przyszła decyzja techniczna i biznesowa musi być z nim spójna. Bez niego projekt "rozlewa się" — dodajesz funkcje, które nie prowadzą do celu.

Co dokument musi zawierać:
  • Elevator pitch — opis produktu w 2 zdaniach (kto, co, po co)
  • Persona użytkownika — kim jest artysta, który z tego korzysta? (wiek, gatunek muzyczny, budżet, umiejętności tech)
  • Problem statement — jaki problem rozwiązujemy? dlaczego obecne rozwiązania nie wystarczają?
  • Zakres MVP — minimum funkcji do pierwszego testu z użytkownikami
  • Out of scope — czego NIE robimy (np. nie generujemy muzyki, nie zarządzamy prawami autorskimi)
  • Success metrics — jak mierzymy, czy to działa? (np. 10 artystów użyło, 50 uploady w miesiąc)

🔧 Dane potrzebne od Ciebie

Żeby stworzyć ten dokument, zapytam Cię m.in. o:

1. Czy sam jesteś artystą/producentem, czy budujesz to dla innych?
2. Jaki gatunek/gatunki muzyczne? (wpływa na wybór platform)
3. Czy masz konto na którejś z platform? (YouTube, SoundCloud, Bandcamp)
4. Budżet na narzędzia (API, hosting, dystrybutor)?
5. Czy chcesz to sprzedawać innym artystom, czy budować dla siebie?

Moduł: Mapa integracji platform

🔌 Platform Integration Map
Cel: dokumentacja techniczna KAŻDEJ platformy — endpointy, limity, kody, auth

To będzie najgrubszy dokument — "podręcznik" do każdej platformy z gotowym kodem w Pythonie. Dla każdej platformy opiszemy: jak się zalogować, jak uploadować, jakie są limity, co może pójść źle, i gotowe snippety kodu.

Platformy Tier 1 — pełne API (budujemy najpierw)

🟢 YouTube Data API v3 — upload wideo

Co udokumentujemy: OAuth 2.0 flow, endpoint videos.insert, quota system (10k jedn./dzień), wymagany audyt Google dla publicznych uploady, obsługa miniaturek, playlisty, opisy SEO, tagi.

Gotowe elementy: Skrypt Python do uploadu, szablony metadanych, kalkulator quota.

🟢 SoundCloud API — upload audio

Co udokumentujemy: OAuth 2.1, endpoint POST /tracks (multipart, do 500MB), tagowanie, playlist management, artwork upload, metadane.

Gotowe elementy: Klasa Python SoundCloudUploader, obsługa błędów, retry logic.

🟢 Internet Archive — upload do kolekcji

Co udokumentujemy: Biblioteka Python "internetarchive", CLI "ia", kolekcje opensource_audio/movies, rate-limiting, best practices unikania blokad.

Gotowe elementy: Skrypt batch upload, konfiguracja metadanych Dublin Core.

Platformy Tier 2 — brak API uploadu (tu potrzebne obejścia)

Dla każdej platformy bez API przygotujemy pełną analizę z rozwiązaniami:

⛔ Problem #1: Spotify / Apple Music / Tidal / Deezer — brak API uploadu
Rozwiązanie A — bezpieczne

Integracja z dystrybutorem (DistroKid / TuneCore)

Używamy DistroKid ($22.99/rok) jako pośrednika. Agent przygotowuje wszystkie metadane, artysta klika "wyślij" → dane lecą na konto DistroKid. Nie budujemy własnego uploadu na platformy streamingowe.

  • Założenie konta DistroKid
  • AI generuje metadane w formacie wymaganym przez DistroKid
  • Artysta loguje się i uploaduje ręcznie (ale z gotowymi danymi)
  • Koszt: $23/rok — znacznie taniej niż własna infrastruktura
Rozwiązanie B — partnerskie

Partnerstwo API z dystrybutorem

Negocjacja z DistroKid, Amuse (darmowy), lub Ditto Music o dostęp do ich wewnętrznego API. Jako "narzędzie, które przysyła im klientów" masz argument negocjacyjny.

  • Napisanie propozycji partnerskiej do 3-5 dystrybutorów
  • Demonstracja MVP jako dowód wartości
  • Negocjacja warunków API access
  • Czas: 2-6 miesięcy na zamknięcie dealu
Rozwiązanie C — własny dystrybutor

Zostanie licencjonowanym dystrybutorem

Podpisanie umów dystrybucyjnych bezpośrednio ze Spotify, Apple, Tidal itd. Wymaga zarejestrowania firmy, prawników, i spełnienia wymagań platform.

  • Rejestracja spółki (sp. z o.o. / LLC)
  • Kontakt z zespołami partnerskimi Spotify for Labels, Apple Music for Artists
  • Spełnienie wymagań: katalog min. X utworów, zabezpieczenia techniczne
  • Koszt: 5k-50k PLN (prawnik + setup), czas: 3-12 mies.
☠️ Rozwiązanie ostateczne

Reverse-engineering API dystrybutora + browser automation

Przechwycenie komunikacji sieciowej DistroKid (devtools → Network tab), odtworzenie ich wewnętrznych requestów API, i automatyzacja uploadu przez Playwright z emulacją ludzkiego zachowania.

  • Analiza requestów HTTP w panelu DistroKid
  • Playwright z fingerprint spoofing (stealth plugin)
  • Randomizowane opóźnienia, losowe ruchy myszą
  • Rotacja IP przez proxy residencjalne
  • ⚠️ Łamie ToS DistroKid → ban konta, utrata katalogu muzyki
  • ⚠️ Potencjalne naruszenie CFAA (Computer Fraud and Abuse Act)
⛔ Problem #2: Bandcamp — API tylko dla labeli
Rozwiązanie A — prepopulacja

AI przygotowuje pakiet "Bandcamp-ready"

Agent generuje: tytuły, opisy, tagi, cenę sugerowaną, okładkę w dokładnym formacie Bandcamp (1400x1400px). Artysta dostaje gotowy pakiet z instrukcją i uploaduje ręcznie w 2 minuty.

Rozwiązanie B — label proxy

Założenie labelu / konta labelowego

Rejestracja jako "label" na Bandcamp daje dostęp do API. Artyści "podpisują się" pod Twój label, upload idzie przez API.

  • Rejestracja labelu na Bandcamp (bezpłatne)
  • Kontakt z Bandcamp o API access
  • Umowa z artystami: licencja do dystrybucji pod labelem
Rozwiązanie C — Chrome Extension

Rozszerzenie przeglądarki artysty

Budowa Chrome Extension, które artysta instaluje. Extension auto-wypełnia formularze Bandcamp danymi z AI. Działa w przeglądarce ARTYSTY, nie na serwerze → mniejsze ryzyko bana.

☠️ Rozwiązanie ostateczne

Headless browser z sesją artysty

Artysta podaje credentials → Playwright loguje się na jego konto Bandcamp → automatycznie wypełnia formularz uploadu → publikuje. Pełna emulacja ludzkiego przeglądarki.

  • Playwright + stealth plugin (undetected-playwright)
  • Emulacja device fingerprint artysty
  • Przechowywanie credentials (szyfrowanie AES-256)
  • ⚠️ Przechowywanie cudzych haseł = ogromne ryzyko bezpieczeństwa
  • ⚠️ Łamie ToS Bandcamp + potencjalne naruszenia RODO
⚠️ Problem #3: TikTok / Instagram / Facebook — ryzykowne API
Rozwiązanie A — content kit

Generowanie gotowych pakietów do ręcznego wrzucenia

AI tworzy: gotowe klipy wideo (15s, 30s, 60s), tekst postu, hashtagi, najlepszą porę publikacji. Artysta dostaje ZIP z materiałami + instrukcję.

Rozwiązanie B — oficjalne schedulery

Integracja z Buffer / Hootsuite / Later API

Te narzędzia MAJĄ oficjalne API do planowania postów. Nie uploadujesz bezpośrednio na TikTok — przesyłasz do Buffer, który to robi za Ciebie legalnie.

  • Buffer API (darmowy plan: 3 kanały)
  • Hootsuite API (od $49/mies., ale silniejsze)
  • Meta Business Suite API (oficjalne, dla FB + IG)
  • TikTok Content Posting API (wymaga konta biznesowego)
Rozwiązanie C — TikTok Creator Tools

TikTok Content Posting API (oficjalne)

TikTok ma oficjalne API do postowania, ale wymaga: konta deweloperskiego, weryfikacji biznesowej, review aplikacji. Proces trwa 2-4 tygodnie.

☠️ Rozwiązanie ostateczne

Appium + emulator Android / ADB

Emulacja telefonu z zainstalowanym TikTokiem/Instagramem. Bot działa jak człowiek: otwiera aplikację, klika "dodaj wideo", uploaduje, wpisuje opis, publikuje.

  • Android emulator (Genymotion / Android Studio AVD)
  • Appium / ADB do sterowania aplikacjami mobilnymi
  • Frida framework do obejścia detekcji rootowania
  • Losowe wzorce interakcji (human-like delays)
  • ⚠️ TikTok i Instagram aktywnie wykrywają emulatory
  • ⚠️ Permanentny ban konta jest niemal pewny

Moduł: AI Pipeline

🧠 AI Pipeline — serce systemu
Cel: opisać JAK ai generuje wartość na każdym etapie

To jest ten element, który odróżnia nasz produkt od prostego skryptu do uploadu. AI analizuje muzykę i tworzy kompletny pakiet promocyjny. Dokument będzie zawierał gotowe prompty, które wystarczy skopiować.

Pipeline składa się z 6 kroków:
  • Krok 1 — Analiza audio: Rozpoznanie gatunku, tempa (BPM), nastroju, kluczowych instrumentów (biblioteka librosa / Essentia)
  • Krok 2 — Generowanie metadanych: LLM tworzy tytuł (jeśli brak), opis SEO, tagi (osobne dla każdej platformy), kategorie
  • Krok 3 — Generowanie okładki: Na podstawie gatunku i nastroju → prompt do DALL-E / Midjourney / Stable Diffusion → okładka w 3 wariantach
  • Krok 4 — Content kit social media: Gotowe posty (tekst + hashtagi), klipy wideo (15/30/60s), stories, Reels
  • Krok 5 — Strategia release: AI sugeruje dzień/godzinę publikacji, kolejność platform, plan promocji na 7 dni
  • Krok 6 — Analityka post-release: Zbieranie danych po publikacji, raport co zadziałało, sugestie na przyszłość
🧩 Problem #4: Który model AI wybrać do generowania metadanych?
Rozwiązanie A — Claude API (Anthropic)

Koszt: ~$3/mln tokenów wejściowych (Sonnet). Najlepszy w zrozumieniu kontekstu muzycznego i generowaniu kreatywnych tekstów po polsku. Bardzo dobre trzymanie się instrukcji.

Rozwiązanie B — OpenAI GPT-4o

Koszt: ~$2.50/mln tokenów. Dobry w generowaniu SEO, więcej tutoriali online. Silny w tworzeniu hashtagów i krótkich treści marketingowych.

Rozwiązanie C — lokalny model (Llama 3 / Mistral)

Koszt: $0 (po pobraniu). Wymaga karty graficznej 8GB+ VRAM. Pełna prywatność, brak limitów API, ale niższa jakość outputu i trudniejsza konfiguracja.

☠️ Rozwiązanie ostateczne

Bezpłatne korzystanie z komercyjnych API przez reverse proxy

Istnieją publiczne reverse proxy do API OpenAI/Claude (np. projekty open-source na GitHub). Daje darmowy dostęp do modeli premium, ale: niestabilne, nielegalne (kradzież compute), ryzyko wycieku danych wejściowych.

🧩 Problem #5: Jak generować okładki albumów?
Rozwiązanie A — DALL-E 3 / GPT Image

API od OpenAI. ~$0.04-0.08 za obraz. Łatwa integracja, dobre wyniki, nie wymaga własnej infrastruktury.

Rozwiązanie B — Stable Diffusion (lokalnie)

Darmowe, uruchamiane na własnym GPU (min. 6GB VRAM). Pełna kontrola nad stylami. Modele fine-tunowane na okładki muzyczne dostępne na Civitai/HuggingFace.

Rozwiązanie C — Canva API + szablony

Canva ma API do generowania grafik z szablonów. Mniej kreatywne, ale przewidywalne i profesjonalne. Koszt: $12/mies. (Canva Pro).

☠️ Rozwiązanie ostateczne

Scraping cudzych okładek + AI style transfer

Pobranie okładek z podobnych gatunków → przepuszczenie przez model style transfer → "inspirowana" okładka. Technicznie to derivative work, które może naruszać prawa autorskie oryginału.

Moduł: Strategia obejścia ograniczeń

⚡ Constraint Bypass Strategy
Cel: dla każdego "muru" znaleźć drzwi, okno, i dynamit

Ten dokument zbiera WSZYSTKIE blokady zidentyfikowane w raporcie i dla każdej daje pełne spektrum rozwiązań. To kluczowy dokument decyzyjny — pomaga wybrać ścieżkę dla każdego problemu.

🔐 Problem #6: YouTube wymaga audytu Google do publicznych uploadów
Rozwiązanie A — przejście audytu

Złożenie aplikacji do Google API compliance review. Przygotowanie: privacy policy, wyjaśnienie use-case, demo aplikacji. Czas: 2-6 tygodni.

  • Stworzenie strony internetowej projektu (landing page)
  • Napisanie Privacy Policy i Terms of Service
  • Nagranie demo wideo pokazującego jak aplikacja działa
  • Wypełnienie formularza w Google Cloud Console
  • Odpowiadanie na pytania reviewerów (w j. angielskim)
Rozwiązanie B — YouTube Studio jako fallback

Artysta uploaduje ręcznie przez YouTube Studio, ale AI przygotowuje: tytuł, opis, tagi, miniaturkę, ustawienia publikacji. Agent kopiuje to do schowka / generuje plik konfiguracyjny.

Rozwiązanie C — Unlisted → Public workflow

Uploaduj jako "unlisted" (niepubliczny) przez API (nie wymaga audytu) → artysta ręcznie zmienia na "public" jednym klikiem w YouTube Studio.

☠️ Rozwiązanie ostateczne

Selenium z zalogowaną sesją YouTube

Przechwycenie cookies sesji artysty → Selenium wchodzi na studio.youtube.com → emuluje upload ręczny.

  • Export cookies z przeglądarki (extension "EditThisCookie")
  • Selenium / Playwright ładuje cookies
  • Emulacja kroków uploadu: wybierz plik → tytuł → opis → publikuj
  • ⚠️ Google aktywnie wykrywa automatyzację — CAPTCHA, weryfikacja
⏱️ Problem #7: Rate-limiting i quotas na wszystkich platformach
Rozwiązanie A — kolejkowanie inteligentne

System kolejki z respektowaniem limitów

Celery + Redis zarządza uploadami. Każda platforma ma skonfigurowany limit requestów. System automatycznie rozłożone uploady w czasie.

Rozwiązanie B — multi-konto (pule API keys)

Rejestracja wielu projektów Google Cloud / kont deweloperskich. Rotacja kluczy API zwiększa łączny limit. Legalne, ale wymaga zarządzania wieloma kontami.

Rozwiązanie C — priorytetyzacja platform

Nie uploaduj na wszystkie platformy jednocześnie. Kolejność: SoundCloud (natychmiast) → YouTube (w ciągu godziny) → Internet Archive (następnego dnia). Rozkłada obciążenie naturalnie.

☠️ Rozwiązanie ostateczne

Masowa rotacja proxy + kluczy API

Kupno setek kluczy API (farma kont Google) + proxy residencjalne do obejścia rate-limitów per-IP. Symulacja wielu niezależnych użytkowników.

  • Automatyczna rejestracja kont Google (reCAPTCHA solver)
  • Proxy residencjalne (Bright Data / SmartProxy, ~$15/GB)
  • Rotacja klucza API co N requestów
  • ⚠️ Łamie ToS Google, fraud detection wykrywa farmy
🪪 Problem #8: Weryfikacja tożsamości artysty na platformach
Rozwiązanie A — instrukcja krok-po-kroku

AI generuje checklist weryfikacji dla każdej platformy: jakie dane podać, jakie zdjęcia przygotować, jak wypełnić formularz. Artysta robi to sam, ale z "asysentem".

Rozwiązanie B — weryfikacja przez dystrybutora

DistroKid automatycznie weryfikuje artystę na Spotify for Artists i Apple Music. Używając dystrybutora, omijasz ręczną weryfikację.

Rozwiązanie C — agent concierge

AI prowadzi artystę przez weryfikację w trybie konwersacji: "teraz otwórz tę stronę, kliknij tu, wpisz to". Jak osobisty asystent, ale nie robi tego za niego.

☠️ Rozwiązanie ostateczne

Automatyczna weryfikacja z danymi artysty

Artysta podaje pełne dane (dowód, zdjęcie, adres) → bot wypełnia formularze weryfikacji automatycznie.

  • OCR na dowodzie osobistym (Tesseract / Google Vision)
  • Auto-fill formularzy Selenium
  • ⚠️ Przechowywanie skanów dowodów = ekstremalnie wrażliwe dane (RODO art. 9)
  • ⚠️ Odpowiedzialność prawna za wyciek danych osobowych

Moduł: Model biznesowy

💰 Business Model
Cel: jak na tym zarobić (lub przynajmniej pokryć koszty)

Dokument opisze modele monetyzacji, wyliczenie kosztów operacyjnych, i punkt break-even (kiedy przestajesz dokładać).

Co dokument będzie zawierał:
  • Model freemium: darmowy tier (1 release/mies.) + płatny ($5-15/mies.)
  • Model pay-per-release: $2-5 za każde wydanie (jednorazowo)
  • Koszty stałe: hosting (~$20/mies.), API AI (~$30/mies. przy 100 użytkownikach), domena ($12/rok)
  • Break-even: szacunkowo 30-50 płacących użytkowników
  • Kanały pozyskania: Reddit (r/WeAreTheMusicMakers), Discord muzyczne, TikTok o produkcji
  • Ryzyka: churn (odejście użytkowników), zmiana API platform, konkurencja
